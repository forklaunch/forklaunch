# Migrating to `wrapEmWithTenantContext`

`@forklaunch/core/persistence` now ships a single helper that fixes a
silent, hard-to-diagnose class of decryption failures on encrypted columns
when using a pooled database driver (e.g. `pg`).

If your project was generated by an older version of the CLI, your
`registrations.ts` looks like this:

```ts
EntityMgr: {
  lifetime: Lifetime.Scoped,
  type: EntityManager,
  factory: (
    { Orm },
    context: { entityManagerOptions?: ForkOptions; tenantId?: string }
  ) => {
    const em = Orm.em.fork(context.entityManagerOptions);
    if (context.tenantId) {
      em.setFilterParams('tenant', { tenantId: context.tenantId });
      // possibly: setEncryptionTenantId(context.tenantId);
    }
    return em;
  }
}
```

You should replace it with:

```ts
import { wrapEmWithTenantContext } from '@forklaunch/core/persistence';

EntityMgr: {
  lifetime: Lifetime.Scoped,
  type: EntityManager,
  factory: (
    { Orm },
    context?: { entityManagerOptions?: ForkOptions; tenantId?: string }
  ) =>
    wrapEmWithTenantContext(
      Orm.em.fork(context?.entityManagerOptions),
      context?.tenantId
    )
}
```

That's the entire change. No call sites in your services / controllers
need to be updated.

## Why this change matters

If your entities use `EncryptedType` (any column declared with
`fp.string().compliance('pii' | 'phi' | 'pci')`), the previous factory
had a silent footgun under load:

1. Your handler calls `emFactory({ context: { tenantId: orgA } })`.
2. The factory calls `setEncryptionTenantId(orgA)`, which uses
   `AsyncLocalStorage.enterWith({ tenantId: orgA })`.
3. Your handler does `em.find(SomeEntity, ...)` which populates an
   encrypted column.
4. MikroORM dispatches the SELECT through the `pg` connection pool. The
   pool grabs an idle connection (long-lived, created at pool init).
5. The query resolves on the connection. Node fires the resolution
   callback **inside the pool connection's async resource**, not the
   handler's.
6. MikroORM hydration runs there and calls
   `getCurrentTenantId()` → reads the connection resource's store →
   gets `''` (or some stale value from a previous request). The wrong
   FieldEncryptor key is derived → `DecryptionError`.

`enterWith` mutates the *current* async resource's store. Promise
continuations in your handler chain inherit it (good), but resumption
callbacks fired by long-lived foreign async resources do not (bad). For
single-tenant local development this almost always happens to work
because the pool's first connection is created lazily inside your one
request, so the connection resource accidentally inherits the right
tenant. Multi-tenant production with a warm pool reproduces the failure
constantly.

## What `wrapEmWithTenantContext` does

It returns a `Proxy` over the forked `EntityManager` whose `get` trap
wraps every method in `withEncryptionContext(tenantId, fn)`.
`withEncryptionContext` uses `AsyncLocalStorage.run` (not `enterWith`),
which creates a fresh async resource bound to the store. Node's promise
hooks propagate that store forward through pool callback async
resources, so `getCurrentTenantId()` returns the correct value at
hydration time regardless of which pooled connection serviced the
query.

Net effect: every `find / findOne / populate / flush / persist /
nativeUpdate` etc. on the returned EM runs inside a properly bound ALS
scope. There is no per-call-site work; the proxy is the single point of
truth for tenant scoping.

It also still calls `em.setFilterParams('tenant', { tenantId })` so the
existing MikroORM filter that adds `WHERE organization_id = :tenantId`
to your queries continues to work.

If you pass `tenantId === undefined`, the helper returns the bare EM
unwrapped. This is the correct behavior for super-admin / lookup paths
that need to query across tenants before they can determine which one
applies (for example, the typical "load the application by id, then
fork an EM scoped to its owning org" pattern).

## Detecting whether you have any pre-existing bad data

Before this fix, any encrypted-column row that was *written* under the
buggy ALS path was wrapped under the wrong (typically empty) tenant
key. After applying the fix, *reads* of those rows under the correct
tenant will fail.

To check, you can run something like the following inside your service
container (Node, with `@forklaunch/core` installed):

```ts
import { FieldEncryptor } from '@forklaunch/core/persistence';

const enc = new FieldEncryptor(process.env.ENCRYPTION_KEY!);

// For each row of an encrypted column you care about:
const peeled = (() => {
  for (const candidate of ['', applicationOwningOrgId, /* …other orgs */]) {
    try { return { tenant: candidate, value: enc.decrypt(row.value, candidate) }; }
    catch { /* try next */ }
  }
  return null;
})();
```

If most/all of your existing rows decrypt under the empty-string tenant
instead of their owning org, you'll need a one-time re-encryption pass
that peels each row with the empty tenant and rewraps it under the
correct org id. The right "owning org id" depends on your data model
(usually `app.organizationId`, `release.application.organizationId`,
or the row's own `organization_id` column, depending on the table).

## Behavior of `setEncryptionTenantId` after this change

`setEncryptionTenantId` still exists and still uses `enterWith`. The
factory continues to call it as a best-effort seed for synchronous code
paths that read the tenant before invoking an EM method. **It is not
load-bearing for hydration.** The Proxy is.

Application code should still not call `setEncryptionTenantId`
directly. If you need to run a block of code under a specific tenant
without going through an EM, use `withEncryptionContext(tenantId,
fn)`.

## Migration checklist

1. Bump `@forklaunch/core` to a version that exports
   `wrapEmWithTenantContext` (any release after this change).
2. Replace your `EntityMgr` (or `EntityManager`) factory body with
   `wrapEmWithTenantContext(Orm.em.fork(context?.entityManagerOptions),
   context?.tenantId)`.
3. Add the import:
   `import { wrapEmWithTenantContext } from '@forklaunch/core/persistence';`
4. Make `context` optional (`context?: {...}`) — this matches the rest
   of the framework and removes a class of "unscoped lookup" call site
   warnings.
5. Restart any long-running processes (workers, dev servers) so their
   in-memory MikroORM state is reset and they pick up the new factory.
6. If you see decrypt errors after deploying, run the diagnostic above
   to determine whether you have historically-corrupt data, and write a
   one-shot re-encryption migration scoped to the affected tables.

If you generated your project with the CLI, regenerating
`registrations.ts` (or running the database infra modifier) will
produce the new shape automatically.
